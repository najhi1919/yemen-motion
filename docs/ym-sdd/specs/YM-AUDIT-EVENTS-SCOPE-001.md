# YM-AUDIT-EVENTS-SCOPE-001

## نوع الوثيقة وحالتها

- النوع: وثيقة نطاق معمارية وتنفيذية.
- الحالة: مقترح جاهز للمراجعة والاعتماد قبل أي تنفيذ برمجي.
- النطاق: Audit / Tracking للأحداث الأمنية والإدارية والتشغيلية المهمة.

لا تنشئ هذه الوثيقة جدولًا أو خدمة أو واجهة، ولا تعتمد وحدها أسماء حقول قاعدة البيانات أو سياسة الاحتفاظ النهائية.

## الهدف

تحديد نطاق نظام Audit / Tracking في منصة Yemen Motion قبل تنفيذ أي جداول أو APIs أو خدمات تسجيل فعلية، بما يضمن أن يبدأ حفظ الأحداث المهمة مبكرًا بطريقة منظمة وآمنة وقابلة للمراجعة.

## القرار

- يبدأ المشروع بتأسيس تسجيل الأحداث قبل بناء واجهات Reports وAnalytics الكاملة.
- الهدف هو حفظ الأحداث المهمة من الآن حتى تتوفر لاحقًا مادة تاريخية موثوقة للتدقيق والتقارير والتحليلات.
- لا يعني «تسجيل كل شيء» نسخ كل payload أو تخزين الأسرار أو البيانات الحساسة الخام.
- تكون الجهة الخادمة هي المصدر الموثوق لتسجيل الأحداث الحساسة، ولا يعتمد إثبات الحدث الأمني أو الإداري على الواجهة وحدها.
- يبقى Audit / Tracking مصدرًا مساعدًا، ولا يستبدل جداول المجال الأصلية أو أنظمة Reports وAnalytics.

## تعريف المصطلحات

| المصطلح | التعريف |
| --- | --- |
| Audit Log | سجل تدقيق للأحداث المهمة أمنيًا وإداريًا، يجيب عن: من فعل ماذا، ومتى، وعلى أي مورد، وما النتيجة؟ |
| Activity Log | سجل نشاط يمكن عرضه لاحقًا داخل الإدارة أو التقارير لتوضيح الأحداث المفيدة للمراجعة التشغيلية. |
| Tracking Event | حدث استخدام أو تنقل أو تفاعل له غرض محدد، ويساعد لاحقًا على تحسين تجربة المستخدم أو بناء Analytics. |
| Analytics | قراءة البيانات وتجميعها ومقارنتها بعد تراكمها لاكتشاف الاتجاهات والفروق. |
| Reports | بحث تاريخي وتفاصيل قابلة للمراجعة ضمن نوع تقرير ونطاق زمني محددين. |

قد تشترك الأنواع الثلاثة الأولى في بنية تخزين موحدة لاحقًا، لكن الغرض ومستوى الحساسية وسياسة الوصول لكل نوع يجب أن يبقيا واضحين.

## مبادئ التسجيل

1. تسجل الأحداث المهمة والقابلة للمراجعة، لا كل حركة تقنية منخفضة القيمة.
2. يخزن الحد الأدنى اللازم لتحقيق غرض أمني أو تشغيلي أو تحليلي محدد.
3. تنقح الحقول الحساسة قبل الحفظ، وليس بعد وصولها إلى السجل.
4. لا تخزن كلمات المرور أو التوكنات أو الكوكيز أو أسرار الدفع أو مفاتيح الوصول.
5. لا تخزن payloads كاملة بصورة عشوائية أو احتياطية.
6. تسجل الجهة الخادمة الأحداث الحساسة بعد التحقق من النتيجة الفعلية للعملية.
7. تعامل أحداث الواجهة غير الحساسة، مثل page views، كمدخلات غير موثوقة تحتاج تحققًا وتقليلًا للبيانات عند تنفيذها لاحقًا.
8. تكون السجلات قابلة للبحث والفلترة حسب النوع والتصنيف والفاعل والهدف والنتيجة والوقت، ضمن حدود الصلاحية.
9. يكون لكل حدث نوع ثابت وواضح، وفاعل، وهدف عند وجوده، ونتيجة، وسياق آمن.
10. لا تستخدم السجلات لاتخاذ قرار صلاحية حالي؛ نظام الصلاحيات الفعلي يبقى مصدر قرار السماح أو المنع.

## قواعد تسمية الحدث

- يقترح استخدام أسماء مستقرة بصيغة نقطية مثل `user.login.success` و`role.updated`.
- يصف الاسم مجال الحدث والإجراء أو النتيجة، ولا يتضمن معرفات مستخدمين أو موارد.
- لا يغير معنى اسم الحدث بعد اعتماده؛ يضاف إصدار أو اسم جديد إذا تغير العقد جذريًا.
- توثق metadata المسموحة لكل نوع حدث بدل قبول كائن مفتوح غير مقيد.

## الشكل النظري المقترح للحدث

الحقول الآتية مقترحة لتثبيت العقد المفاهيمي فقط. لا تحدد هذه الوثيقة نوع العمود أو الفهرس أو اسم الجدول النهائي.

| الحقل | الغرض | القيود المقترحة |
| --- | --- | --- |
| `id` | معرف فريد للحدث | يحدد شكله النهائي عند تصميم التخزين. |
| `event_type` | الاسم الثابت للحدث | قيمة واضحة بصيغة نقطية، ولا تحتوي بيانات شخصية. |
| `category` | المجال العام للحدث | قيمة من تصنيف معتمد وقابل للفلترة. |
| `severity` | مستوى أهمية الحدث | إحدى القيم المعتمدة في قسم مستويات الأهمية. |
| `actor_type` | نوع الفاعل | مثل مستخدم أو نظام أو زائر غير موثق؛ دون أسرار مصادقة. |
| `actor_id` | معرف الفاعل عند توفره | يقبل غياب القيمة عند محاولة دخول فاشلة أو حدث نظامي. |
| `actor_role` | لقطة سياقية لدور الفاعل وقت الحدث | ليست مصدرًا لقرار الصلاحية، ولا يفترض أنها تظل محدثة بعد تغير الدور. |
| `target_type` | نوع المورد المتأثر | مثل مستخدم أو دور أو صلاحية، ويقبل غياب القيمة. |
| `target_id` | معرف المورد المتأثر | يخزن عند وجود هدف محدد ومسموح بتسجيله. |
| `action` | الفعل الموحد | مثل إنشاء أو تحديث أو حذف أو عرض أو بحث أو منع. |
| `outcome` | نتيجة الحدث | مثل نجاح أو فشل أو منع، بقيم محددة وليست رسالة استثناء خام. |
| `ip_address` | سياق شبكي أمني اختياري | بيانات شخصية محتملة؛ تخضع للتقليل والحماية وسياسة الاحتفاظ. |
| `user_agent` | سياق تقني اختياري للعميل | ينقح ويحدد طوله، ولا يستخدم لبناء بصمة غير ضرورية للمستخدم. |
| `request_id` أو `correlation_id` | ربط أحداث الطلب أو العملية الواحدة | لا يكون session id أو token، ولا يكشف سرًا. |
| `metadata` | سياق إضافي منقح | allowlist خاصة بكل event type، وليست payload مفتوحة. |
| `occurred_at` | وقت وقوع الحدث فعليًا | يميز وقت الحدث عن وقت إدخاله إلى التخزين. |
| `created_at` | وقت إنشاء سجل الحدث | يستخدم لمتابعة تأخير الحفظ أو الإدخال عند الحاجة. |

## البيانات التي يمنع تسجيلها

تمنع القيم التالية في الحقول المباشرة وداخل metadata ورسائل الأخطاء المنسوخة:

- `password`
- `password_confirmation`
- `current_password`
- `token`
- `access_token`
- `refresh_token`
- `cookie`
- session id الخام
- بيانات بطاقات الدفع
- الأسرار ومفاتيح التشفير
- API keys
- الملفات الخام دون غرض واضح ومعتمد
- payload كامل دون تنقيح

يشمل المنع القيم نفسها، لا أسماء الحقول فقط. لذلك يجب أن يسبق الحفظ redaction مركزي، وألا يعتمد المنع على حذف مفاتيح معروفة بعد تخزين الحدث.

## metadata الآمنة

- تعتمد allowlist منفصلة قدر الإمكان لكل `event_type`.
- ترفض أو تتجاهل الحقول غير المعروفة وفق قرار يثبت في مرحلة foundation.
- لا تحتوي metadata على نسخ كاملة من Models أو Requests أو Exceptions.
- تحول أسباب الفشل إلى رموز موحدة وآمنة بدل حفظ رسائل قد تتضمن SQL أو مسارات أو بيانات مستخدم.
- تخزن الفروق الإدارية اللازمة فقط، مع تنقيح القيم الحساسة وتحديد حجم البيانات.
- لا يحفظ نص بحث Dashboard الخام افتراضيًا؛ يفضل حفظ نوع البحث والمرشحات والنطاق وعدد النتائج ما لم يعتمد غرض واضح للنص.
- لا تحفظ query strings أو أجزاء URL التي قد تحتوي رموزًا أو معرفات حساسة ضمن page views.

## الخصوصية والأمان

### تقليل البيانات والتنقيح

- ينفذ redaction قبل الحفظ لجميع المسارات التي تنتج أحداثًا.
- تعامل `ip_address` و`user_agent` كبيانات قد تكون شخصية، ولا تجمع إلا عند وجود غرض أمني أو تشغيلي واضح.
- يحدد لاحقًا ما إذا كانت عناوين IP ستخزن كاملة أو مختصرة أو مشتقة، وفق الحاجة وسياسة الخصوصية.
- تحدد حدود الطول والقيم المسموحة للحقول النصية حتى لا يتحول السجل إلى مخزن بيانات غير منضبط.

### مصدر الحدث والثقة

- تسجل الجهة الخادمة نجاح أو فشل العمليات الحساسة بعد معرفة النتيجة الفعلية.
- لا تقبل الواجهة تحديد `actor_id` أو `outcome` الموثوق للأحداث الأمنية.
- تحتاج أحداث Tracking القادمة من العميل إلى تحقق ومعدل إرسال محدود وعقد metadata ضيق عند تنفيذها لاحقًا.

### الوصول والمراجعة

- تكون قراءة السجلات محصورة في صلاحيات إدارية دقيقة ستحدد قبل بناء أي API أو UI لها.
- لا يمنح وجود السجل حقًا تلقائيًا في رؤية بيانات المورد المرتبط به.
- يجب منع التعديل الاعتيادي على أحداث التدقيق بعد حفظها، مع تحديد آلية التصحيح والاحتفاظ بصورة منفصلة في التصميم اللاحق.
- يراجع الوصول إلى سجلات التدقيق نفسها ضمن سياسة أمنية لاحقة، دون افتراض أن كل موظف داخلي يستطيع قراءتها.

### الغرض المشروع

- السجلات أداة للأمن والتشغيل والمراجعة، وليست أداة تجسس على المستخدمين.
- تسجل page views داخل الإدارة للصفحات المهمة فقط، ولغرض أمني أو تحسيني محدد.
- لا ينشأ تتبع شامل للصفحات العامة أو سلوك المستخدمين دون حاجة موثقة ومراجعة خصوصية مستقلة.

### الاحتفاظ والحذف

- يجب تحديد سياسة retention قبل تشغيل التخزين في الإنتاج.
- تفرق السياسة بين الأحداث الأمنية والإدارية وأحداث Tracking منخفضة الحساسية أو القيمة.
- تحدد السياسة مدة الاحتفاظ، وآلية الأرشفة أو الحذف، والاستثناءات النظامية، ومن يملك تنفيذها.
- لا تعني قابلية التدقيق الاحتفاظ غير المحدود بكل حدث.

## البحث والفلترة المستقبلية

يجب أن يسمح العقد لاحقًا، ضمن صلاحيات القراءة، بالفلترة حسب:

- `event_type`
- `category`
- `severity`
- `actor_type` و`actor_id`
- `target_type` و`target_id`
- `action`
- `outcome`
- نطاق `occurred_at`
- `request_id` أو `correlation_id`

لا تتضمن هذه الوثيقة API بحث أو فهارس قاعدة بيانات؛ يحدد ذلك بعد اعتماد حجم البيانات وأنماط الاستعلام الفعلية.

## العلاقة مع Reports وAnalytics

- تستخدم Reports سجلات audit/activity لعرض الأحداث التاريخية والتغييرات الإدارية القابلة للمراجعة.
- قد تستخدم Analytics الأحداث بصورة مجمعة لاحقًا عند وجود سؤال تحليلي واضح وحجم بيانات مناسب.
- تبقى الجداول الأصلية مصدر الحقيقة لحالة المستخدم أو الطلب أو العمل أو المعاملة المالية.
- يوضح Audit من نفذ الإجراء ومتى وعلى ماذا وما النتيجة، ولا يستبدل سجل المجال الأصلي.
- لا تعتمد التحليلات طويلة المدى على metadata غير مستقرة؛ تحتاج الأحداث المرشحة للتحليل إلى عقد ثابت أو تجميع موثق.

## أول تنفيذ برمجي مقترح بعد اعتماد الوثيقة

المهمة التالية المقترحة هي:

`YM-AUDIT-EVENTS-FOUNDATION-001`

ويشمل نطاقها المقترح لاحقًا، بعد مراجعته واعتماده بصورة مستقلة:

- migration لجدول `audit_events` أو `activity_events` بعد حسم الاسم والعقد.
- Model يمثل الحدث دون كشف بيانات محظورة.
- Service مركزية لتسجيل الأحداث وتنقيح metadata.
- helper أو action محدود لتسجيل الحدث من Controllers أو طبقة المجال.
- tests تثبت الحفظ والتنقيح ومنع الحقول الحساسة وحدود metadata.

لا تعد هذه القائمة إذنًا بتنفيذ المكونات ضمن مهمة النطاق الحالية.

## قرارات يجب حسمها في مرحلة Foundation

- الاسم النهائي للجدول والنموذج، وهل تكفي بنية موحدة لكل الأنواع.
- أنواع المعرفات والفهارس المطلوبة بناءً على الاستعلامات المتوقعة.
- سياسة التعامل مع فشل تسجيل حدث حساس: متى تفشل العملية الأصلية، ومتى يسجل فشل ثانوي فقط.
- التخزين المتزامن أو غير المتزامن لكل فئة حدث، وضمان ترتيب `occurred_at` و`created_at`.
- سياسة IP وuser agent والاحتفاظ حسب البيئة والغرض.
- الصلاحيات الدقيقة لقراءة السجلات أو تصديرها مستقبلًا.
- قواعد منع التكرار عند إعادة الطلب أو تنفيذ job أكثر من مرة.

## خارج النطاق الآن

- لا migration.
- لا Model.
- لا Service أو helper أو action.
- لا API للكتابة أو القراءة أو البحث.
- لا UI لعرض الأحداث.
- لا تغييرات على Dashboard.
- لا Reports UI.
- لا Analytics UI.
- لا export إلى PDF أو Excel.
- لا snapshots أو aggregations.
- لا تعديل على الصلاحيات أو routes.
